tinker — Privacy & Data Protection

Privacy Is a System Responsibility

Privacy is not a statement we publish. It is a boundary we enforce.

Every institution operating on tinker entrusts us with sensitive data — student records, learning patterns, institutional operations. This data is not incidental to the system. It is central to it. And because it is central, it is governed with strict intent — at every stage, without exception.

We do not treat data as an asset. We treat it as a responsibility. That distinction is not semantic. It determines every architectural decision we make about how data moves, where it lives, who can see it, and when it is gone.

DataAccessControlGovernance
Our Position on Data

Our Position on Data

One principle governs everything that follows:

Data belongs to the institution that generates it.

Ownership
Institution→owns
tinker→processes

Student data belongs to the school. Institutional data belongs to the institution. Parent-linked data remains tied to the student identity it was created to serve.

tinker does not claim ownership over institutional data. tinker does not use institutional data as a commercial asset. tinker does not extract value from data outside the system's defined purpose. We operate as a processor and infrastructure provider — not a data owner. That role comes with obligations, not privileges.

Data Collection — Minimal and Purpose-Bound

We collect only what the system requires to function.

Purpose-bound filter

That includes student identifiers — name, class, institutional ID. Learning interaction data — progress, assessments, performance signals. Administrative and operational records. Parent-linked access credentials, for visibility purposes only.

We do not collect unnecessary personal data, behavioural tracking unrelated to learning, external browsing data, or sensitive personal identifiers beyond institutional need. Every data point collected exists for a defined system function. If it does not serve that function, it is not collected. There is no secondary accumulation.

Intent Purpose boundary — no branching
Purpose Limitation

Purpose Limitation

Data within tinker is processed strictly within its intended purpose: delivering personalised learning experiences, enabling institutional management and reporting, supporting AI-driven diagnostics and recommendations, and providing controlled visibility to authorised roles.

Data is not repurposed. Not for advertising. Not for profiling outside the educational context. Not for resale. Not for external analytics unrelated to institutional use. There is no secondary use of institutional data — because there is no authorisation for one.

Data Storage and Infrastructure Control

All data is stored within controlled infrastructure environments.

School ASchool BSchool CSchool D

Institution-specific or logically isolated storage — no cross-institutional data mixing, no shared exposure layers. Data is encrypted at rest. Data is encrypted in transit. Access to stored data is not open. It is governed by strict access policies and continuous system validation.

The data of one institution is structurally inaccessible to another. That is not a configuration. It is the architecture.

Access Control and Data Visibility

Data access is defined by role and responsibility — and nothing else.

Students access only their own learning data. Teachers access data of assigned students. Administrators access institution-wide operational data within their defined scope. Parents access read-only data tied to their child. There is no unrestricted data visibility within the system at any role level.

Every access request is authenticated, authorised, and logged. Implicit access does not exist. All access is explicitly granted and continuously validated — not issued once and left unchecked.

AI and Data Usage

AI systems within tinker operate under strict data boundaries.

AI processes institutional data only within the institution's own environment. There is no cross-institutional data pooling. There is no external model training using institutional data. There is no data export to third-party AI systems.

AI uses data for a defined set of purposes: to understand learning patterns, deliver personalised pathways, and identify gaps and recommend interventions. AI does not use data to train external models, generate unrelated outputs, or build datasets outside institutional control.

The data stays within the governance boundary it was created under. That boundary does not flex to accommodate system convenience.

Data Retention and Lifecycle

Data is not stored indefinitely without purpose.

CollectUseArchiveDelete

Retention is governed by institutional agreements, operational necessity, and regulatory expectations. Data is retained only as long as required. It is archived under controlled conditions when necessary. It is deleted securely upon institutional request or upon contract termination.

There is no indefinite accumulation. Data that no longer serves its defined function does not remain in the system.

Data Deletion and Exit Control

Institutions retain full authority over their data lifecycle — including its end.

Upon termination of services: data can be exported securely upon request, data is permanently deleted from active systems, and backup data is removed under controlled deletion cycles. Deletion is not partial. It is complete and irreversible within defined timelines.

When an institution leaves, their data does not remain with us. That is not a policy concession. It is a structural commitment.

Cookies, Tracking, and Analytics

tinker uses minimal, controlled tracking mechanisms.

Essential On (required)

On the website: essential cookies for functionality, and analytics tools to improve user experience. Behavioural tracking is limited to website interaction — it does not extend to student learning data at any point.

Within the platform: no advertising trackers, no third-party behavioural tracking systems, no external data sharing. All tracking is purpose-specific, transparent, and non-intrusive. Users are informed and can control cookie preferences where applicable.

Contract Security Scope validation
Third-Party Access

tinker does not provide unrestricted third-party access to data.

Third-party involvement is limited to infrastructure providers — hosting, security layers — and essential system integrations where operationally required. All third-party access is contractually governed, security-audited, and restricted to the necessary scope.

There is no open API access exposing sensitive data without strict authentication and system-level control.

Compliance and Regulatory Alignment

Compliance and Regulatory Alignment

tinker operates within Pakistan's regulatory context and in alignment with its emerging data protection frameworks. Our internal standards are built to meet enterprise-grade global expectations — because the institutions we serve require that standard, and the students within them deserve it.

We do not treat compliance as a ceiling. We treat it as a floor.

User Rights and Institutional Control

tinker does not bypass institutional authority.

All data rights are exercised through the institution. Schools control student data access. Schools manage administrative permissions. Schools define retention policies within system capabilities. Parents and users interact with data through institution-defined access models.

This ensures centralised governance, controlled access, and clear institutional accountability. The institution governs. The system serves that governance.

EncryptionAccess controlMonitoringPrivacy
Security as the Foundation of Privacy

Privacy cannot exist without security.

Every privacy guarantee we make is enforced through access control systems, encryption protocols, infrastructure isolation, and continuous monitoring.

Privacy is not dependent on user behaviour or good intent. It is enforced through system design — the same way every other constraint in tinker is enforced. If it is not structural, it is not reliable.

Flows auditedPolicies currentCompliance signalReviews scheduled
Continuous Oversight

Privacy is not static.

Systems are reviewed regularly. Data flows are audited. Policies are updated as requirements evolve and as the system develops.

tinker maintains continuous oversight to ensure data remains protected, usage remains within scope, and the system remains compliant and controlled. The standard does not degrade over time. It is maintained actively.

Trust, Defined by Control

We do not define privacy through promises.

We define it through clear data ownership, controlled system behaviour, enforced access boundaries, and measurable protection standards. Privacy is not declared. It is engineered — and it holds because the system enforces it, not because we ask it to be respected.
